Планирование стека технологий для MMO с **рекурсивным горизонтально-пространственным масштабированием** (вероятно, подразумевается архитектура, подобная *"инстанциированный мир, где инстанции могут порождать под-инстанции"*, или *"бесконечное procedurally generated пространство с динамическим шардингом"*) — это задача архитектуры распределенных систем уровня AAA.

Ниже представлен детальный план стека, разбитый по слоям, с акцентом на отсутствие единой точки отказа (SPOF), детерминизм и возможность масштабирования «до бесконечности» путем добавления обычных серверов.

---

### 1. Концептуальная архитектура: «Рекурсивное масштабирование»

Прежде чем выбирать технологии, нужно определить, как работает «рекурсивность». Я предполагаю архитектуру **Dynamic Space Partitioning**:
*   Мир делится на **Соты (Cells/Chunks)**.
*   Каждая сота — это независимый процесс (Node).
*   Если нагрузка в соте превышает лимит (много игроков, сложная симуляция), сота автоматически **делится (Split)** на 4 дочерние соты (рекурсия).
*   Если игроки уходят, соты **сливаются (Merge)**.
*   Переходы между сотами (границы) должны быть бесшовными для клиента.

---

### 2. Бэкенд (Серверная симуляция)

Это самый критичный слой. Здесь важна производительность, предсказуемость задержек (low latency) и конкурентность.

#### Язык программирования: **Go** или **Rust**
*   **Go (Golang):**
    *   *Плюсы:* Идеальный баланс производительности и скорости разработки. Лучший выбор для **воркеров (Cell Nodes)** и **API Gateway**. Легковесные горутины позволяют держать десятки тысяч активных соединений на одном узле. Простой деплой (один бинарник).
    *   *Минусы:* GC (Garbage Collector) может создавать микро-лаги, но при правильной настройке (без аллокаций в hot path) это не критично для MMO.
*   **Rust:**
    *   *Плюсы:* Абсолютный контроль над памятью, отсутствие GC, детерминизм. Идеален для **тяжелой симуляции физики** или **движка мира** внутри соты, где важна каждая микросекунда.
    *   *Минусы:* Более долгая разработка, сложный онбординг команды.

#### Сетевое взаимодействие: **gRPC + WebRTC (or UDP)**
*   **Сервер-Сервер (East-West трафик):** **gRPC (HTTP/2) на бинарных протоколах (Protobuf)**.
    *   Необходим для общения менеджера сеток (Grid Manager) с воркерами, а также для миграции игроков между сотами.
*   **Клиент-Сервер:**
    *   **Вариант 1 (WebGL/Десктоп):** **WebRTC Data Channel** (если целевая платформа — веб) или **Raw UDP** (для нативного десктопа) с механизмом надежной/ненадежной доставки (аналог RakNet).
    *   *Рекомендация:* Использовать библиотеку **Pion** (Go) или **Netcode.io** (Rust) для стандартизированной модели транспорта.

---

### 3. Оркестрация и сетка сот (The Grid Manager)

Это сердце «горизонтального масштабирования». Нужна система, которая управляет жизненным циклом процессов.

*   **Kubernetes (K8s):**
    *   Базовый оркестратор. Позволяет поднимать тысячи подов (копий симуляции сот).
    *   *Особенность:* Вам понадобится **Custom Scheduler**. Стандартный K8s не умеет автоматически решать, на какой физический нод поставить под «Соту А» чтобы минимизировать задержки между соседними сотами. Нужно писать свой контроллер.
*   **Consul / etcd:**
    *   Используются для **Service Discovery**. Каждая сота (Cell) при старте регистрирует свои координаты (X, Y, Z, Level), свой gRPC эндпоинт и текущую нагрузку.
    *   Менеджер границ (Border Manager) смотрит в Consul, чтобы понять, куда перенаправить игрока, переходящего из клетки A в клетку B.

---

### 4. Базы данных (Data Layer)

В MMO с рекурсивным миром нельзя использовать традиционные реляционные БД в качестве единственного хранилища из-за огромного количества операций записи состояния игроков.

*   **Хранилище состояния (State Storage):** **ScyllaDB** (или Cassandra).
    *   *Почему:* Линейная масштабируемость записи/чтения. Идеально для хранения "слепков" игроков (Player Snapshots) при выходе из игры или миграции между серверами.
    *   *Схема:* Используем `player_id` как партиционный ключ.
*   **Кэш реального времени (Real-time Cache):** **Redis Cluster**.
    *   Используется как общее хранилище для временных данных, которые нужны всем сотам (например, данные гильдии, друзья онлайн, шардированные блокировки).
*   **Персистентный мир (World State):** **PostgreSQL** (асинхронная запись).
    *   Только для экономики, инвентаря (с транзакциями ACID) и аудита. Серверная симуляция не должна дергаться к Postgres напрямую во время игрового цикла.

---

### 5. Поток данных: Message Bus

При рекурсивном делении сот (сотовых узлов) события (например, "игрок произнес сообщение в чат в зоне", "сработало глобальное событие мира") должны доставляться в подписанные узлы без прямого p2p соединения между сотми (это создает граф сложности O(n²)).

*   **NATS JetStream:**
    *   *Альтернатива:* Kafka.
    *   *Роль:* Публикация событий (Event Sourcing). Если сота A разделилась на A1 и A2, она публикует событие `cell.split`. Менеджер сети слушает эту очередь и обновляет маршрутизацию в Consul.

---

### 6. Клиентский стек и синхронизация

Поскольку мир бесшовно масштабируется и рекурсивно делится, клиент должен уметь одновременно «видеть» объекты из нескольких сот (обычно 3x3 сетка вокруг игрока).

*   **Unity (C#) / Unreal Engine (C++):**
    *   Unreal предпочтительнее для сложной сетевой физики и нативных MMO (например, *Albion Online* или *Fortnite*), но требует зрелости команды в C++.
*   **Сетевой клиентский слой:**
    *   Клиент не должен знать, что сервер разделился. Нужна абстракция **"Virtual Connection"**. Клиент держит WebSocket/UDP коннект к шлюзу (Gateway), а шлюз проксирует пакеты в текущую активную соту игрока. Если сота меняется или делится, шлюз обновляет роутинг, не перезапуская соединение клиента.

### 7. Итоговая схема стека технологий

| Компонент | Технология | Обоснование |
| :--- | :--- | :--- |
| **Серверная логика (Cell)** | **Go** (с ECS фреймворком, например, `Arche` или собственная реализация) | Высокая производительность, легковесные потоки (горутины) на соту, простота деплоя. |
| **Тяжелые вычисления (Физика/Пути)** | **Rust** (вынесено в отдельный микросервис) | Если требуется детерминированная симуляция тысяч NPC без GC. |
| **Сервер-Сервер** | **gRPC (Protobuf)** + **NATS** | Быстрые бинарные вызовы для синхронизации границ и асинхронная шина событий для рекурсивных изменений топологии. |
| **Клиент-Сервер** | **WebRTC (Pion)** или **UDP + KCP** | Низкая задержка, поддержка NAT, возможность работы в браузере (WebRTC) и на десктопе. |
| **Оркестрация** | **Kubernetes** + **Custom Operator** | Управление тысячами подов сот. Custom Operator нужен для "affinity" (соседние соты на одном физическом хосте для низкого пинга). |
| **База данных** | **ScyllaDB** (основная) + **Redis** (кэш/локи) | Scylla выдерживает миллионы операций записи состояния в секунду. Redis для мгновенной синхронизации разделения/слияния сот. |
| **Мониторинг** | **Prometheus + Grafana + OpenTelemetry** | Обязательная трассировка (Distributed Tracing) для отладки цепочек "Клиент -> Gateway -> Cell A -> Cell B" при рекурсивном погружении в под-миры. |

---

### 8. Ключевые вызовы (Risks)

1.  **Рекурсивная миграция (Thundering Herd):**
    Если сота делится на 4 части, все игроки внутри нее должны быть моментально перераспределены.
    *   *Решение:* Использовать технику **"Freeze & Copy"**. Старая сота остается в режиме "только чтение" (чтоб игроки не теряли коннект) на 500-1000 мс, пока новые 4 соты загружают состояние из снепшота и подтверждают готовность.
2.  **Консистентность границ:**
    Игрок на границе двух сот может стрелять в игрока из соседней соты.
    *   *Решение:* Паттерн **"Phantom Nodes"** (Фантомы). Каждая сота подписывается на события соседних сот через gRPC стриминг и рендерит удаленные объекты локально, применяя к ним свою физику/коллизии на границе.
3.  **Stateful vs Stateless:**
    Симуляция соты — это stateful приложение. Kubernetes не любит убивать stateful поды без graceful shutdown.
    *   *Решение:* Использовать **Operator Pattern** и **Pod Disruption Budgets**. При удалении пода (обновление версии, деление) контроллер должен инициировать процедуру плавного слияния состояния с соседями.

### Заключение

Рекомендуемый стек: **Go (Rust для физики) + gRPC + NATS + ScyllaDB + Kubernetes + WebRTC/UDP**.

Главная сложность такого проекта лежит не в конкретных языках, а в реализации **Custom Kubernetes Operator**, который будет управлять рекурсивной топологией (сеткой сот), и в реализации **бесшовного пересечения границ** с репликацией состояния между сотами (Phantom Entities).

Отлично. Теперь рассмотрим ключевой момент — **деление одной соты на четыре** (split). Это самая сложная операция в системе, где рекурсивная масштабируемость проявляется во всей красе.

## Сценарий: Достигнут порог деления

### 1. Обнаружение нагрузки

Grid Manager в фоновом режиме постоянно анализирует метрики:

```go
// Grid Manager — циклический анализ
func (gm *GridManager) AnalyzeMetrics() {
    cells := gm.discovery.GetCells()
    
    for _, cell := range cells {
        metrics := gm.prometheus.Query(cell.id, "load_avg_5m")
        
        if metrics.Players > 300 || metrics.Entities > 5000 {
            gm.InitiateSplit(cell)
        }
    }
}
```

Допустим, сота `(0,0,0)` показывает:
- **450 игроков** (порог 300)
- **8200 сущностей** (порог 5000)
- CPU usage: 78%

Grid Manager принимает решение о делении.

---

### 2. Планирование деления

Grid Manager вычисляет новые границы:

```
Исходная сота (level 0): границы [-1000, -1000] до [1000, 1000]

Дочерние соты (level 1):
- (-1,-1,1): [-1000, -1000] до [0, 0]
- ( 1,-1,1): [0, -1000] до [1000, 0]
- (-1, 1,1): [-1000, 0] до [0, 1000]
- ( 1, 1,1): [0, 0] до [1000, 1000]
```

Grid Manager создаёт 4 задания в Kubernetes (CRD):

```yaml
apiVersion: mmo.io/v1
kind: Cell
metadata:
  name: cell_-1_-1_1
spec:
  level: 1
  bounds:
    x_min: -1000
    x_max: 0
    z_min: -1000
    z_max: 0
  parent: cell_0_0_0
```

---

### 3. Подготовка родительской соты

Grid Manager отправляет gRPC команду родительской соте `(0,0,0)`:

```protobuf
message SplitCommand {
    repeated CellDefinition child_cells = 1;
    int32 freeze_window_ms = 2;  // 500ms
}
```

Сота получает команду и переходит в состояние **FROZEN**:

```go
func (c *Cell) PrepareSplit(children []CellDef) {
    // 1. Останавливаем обработку новых подключений
    c.status = CellStatus_FREEZING
    
    // 2. Рассылаем всем клиентам уведомление
    c.Broadcast(&Message{
        Type: "WORLD_FREEZE",
        Data: map[string]interface{}{
            "duration_ms": 500,
            "reason": "world_expanding",
        },
    })
    
    // 3. Сохраняем снепшот состояния
    snapshot := c.ecs.TakeSnapshot()
    c.saveToRedis(snapshot)
    
    // 4. Распределяем сущности по дочерним сотам
    c.partitionEntities(children)
}
```

**Важно:** игроки не теряют соединение, просто перестают получать обновления мира на 500 мс. На клиенте отображается анимация загрузки или "мир замер".

---

### 4. Распределение сущностей (Partitioning)

Родительская сота вычисляет, какой игрок/сущность в какую дочернюю соту попадёт:

```go
func (c *Cell) partitionEntities(children []CellDef) {
    for _, entity := range c.ecs.GetAllEntities() {
        pos := entity.GetComponent(Position)
        
        // Определяем целевую дочернюю соту по координатам
        targetCell := c.findChildByPosition(pos.X, pos.Z, children)
        
        // Сохраняем маппинг в Redis
        redis.HSet("split_migration", entity.ID, targetCell.ID)
        
        // Готовим данные для передачи
        migrationData := entity.Serialize()
        
        // Отправляем в NATS (каждая дочерняя сота получит свою порцию)
        nats.Publish(fmt.Sprintf("cell.migration.%s", targetCell.ID), migrationData)
    }
}
```

**Критический момент:** NPC, которые находятся на границе между двумя будущими сотами, должны быть отданы **только одной** соте. Используется детерминированное правило: например, если координата X или Z == 0, отдаём в соту с положительными координатами (правый/верхний).

---

### 5. Параллельный запуск дочерних сот

Пока родительская сота заморожена, Grid Manager:

1. **Запускает 4 новых пода** (дочерние соты) через Kubernetes.
2. Каждая дочерняя сота при старте:
   - Регистрируется в Consul с уровнем 1 и своими границами
   - Подписывается на свой топик NATS `cell.migration.{cell_id}`
   - Загружает присланные сущности из NATS
   - Инициализирует свою ECS

```go
func (child *Cell) OnStart() {
    // Подписка на миграционные данные
    nats.Subscribe(fmt.Sprintf("cell.migration.%s", child.id), func(msg *nats.Msg) {
        entity := deserialize(msg.Data)
        child.ecs.AddEntity(entity)
    })
    
    // После получения всех сущностей (таймаут или подтверждение от родителя)
    child.readySignal <- true
}
```

Grid Manager ожидает, пока все 4 дочерние соты не сообщат о готовности (через gRPC или NATS).

---

### 6. Атомарное переключение (Cutover)

Самый ответственный момент. Grid Manager использует **двухфазный коммит**:

```go
func (gm *GridManager) CommitSplit(parent Cell, children []Cell) {
    // Фаза 1: PREPARE
    for _, child := range children {
        if !child.Prepare() {  // проверяем, что все сущности загружены
            gm.RollbackSplit(parent, children)
            return
        }
    }
    
    // Фаза 2: COMMIT
    // 1. Обновляем Consul: родительская сота помечается как "split"
    gm.discovery.UpdateCellStatus(parent.ID, CellStatus_SPLIT)
    
    // 2. Отправляем клиентам новые эндпоинты
    players := parent.GetActivePlayers()
    for _, player := range players {
        targetCell := gm.findTargetCell(player.Position, children)
        player.Send(&Message{
            Type: "REDIRECT",
            Data: map[string]interface{}}{
                "new_cell_endpoint": targetCell.GRPCEndpoint,
                "token": generateToken(player),
            },
        })
    }
    
    // 3. Завершаем родительскую соту
    parent.GracefulShutdown()
    
    // 4. Удаляем родительскую регистрацию из Consul
    gm.discovery.DeregisterCell(parent.ID)
    
    // 5. Активируем дочерние соты (разрешаем новые подключения)
    for _, child := range children {
        child.Activate()
    }
}
```

### 7. Клиентская сторона (игрок)

Игрок получает сообщение `REDIRECT`. Клиентский код:

```csharp
// Unity/C# пример
void OnWorldSplit(Message msg) {
    string newEndpoint = msg.Data["new_cell_endpoint"];
    string token = msg.Data["token"];
    
    // Сохраняем локальное состояние (позицию, инвентарь)
    SaveLocalState();
    
    // Закрываем текущее соединение
    networkManager.Disconnect();
    
    // Устанавливаем соединение с новой сотой
    networkManager.Connect(newEndpoint, token);
    
    // После подтверждения соединения отправляем локальное состояние
    // для верификации (anti-cheat)
}
```

**Важно:** Переподключение должно занимать < 100 мс. Игрок видит лишь кратковременную "заморозку" мира (в сумме ~500-600 мс).

---

### 8. Пост-сплит: новые границы

После успешного деления мир выглядит так:

```
Уровень 0: сота (0,0,0) — SPLIT (неактивна, но сохраняется в истории)
Уровень 1: 4 активные соты, каждая управляет своей четвертью мира

Топология в Consul:
- cell_-1_-1_1: [-1000,-1000] до [0,0]
- cell_1_-1_1:  [0,-1000] до [1000,0]
- cell_-1_1_1:  [-1000,0] до [0,1000]
- cell_1_1_1:   [0,0] до [1000,1000]
```

---

### 9. Обработка границ между новыми сотами

Теперь, когда игроки находятся в разных сотах, но могут видеть друг друга на границе, активируется **система фантомов (Phantom Nodes)**:

```go
// Каждая дочерняя сота подписывается на соседние
func (c *Cell) SetupNeighbors() {
    neighbors := c.discovery.GetNeighbors(c.bounds)
    
    for _, neighbor := range neighbors {
        // Устанавливаем gRPC стрим для получения объектов на границе
        stream := neighbor.SubscribeBorderRegion(c.bounds.ExpandedBy(100))
        
        go func(stream pb.CellService_BorderStreamClient) {
            for {
                phantom, _ := stream.Recv()
                c.ecs.AddPhantomEntity(phantom)  // не управляется локально, только рендеринг
            }
        }()
    }
}
```

Клиент получает объекты из нескольких сот одновременно, для него переход бесшовный.

---

### 10. Fallback: если сплит не удался

Grid Manager реализует компенсационную логику:

```go
func (gm *GridManager) RollbackSplit(parent Cell, children []Cell) {
    // 1. Уничтожаем дочерние поды
    for _, child := range children {
        k8s.DeletePod(child.PodName)
    }
    
    // 2. Размораживаем родительскую соту
    parent.Unfreeze()
    
    // 3. Уведомляем клиентов об отмене
    parent.Broadcast(&Message{
        Type: "SPLIT_CANCELLED",
    })
    
    // 4. Логируем инцидент для анализа
    logger.Error("split_failed", "cell", parent.ID)
}
```

## Что произошло с игроками во время сплита

| Время | Событие | Игрок видит |
|-------|---------|-------------|
| 0ms | Нормальная игра | Бесшовный мир |
| T+100ms | Grid Manager решает сплит | — |
| T+200ms | Родительская сота заморожена | "Загрузка мира..." (анимация) |
| T+300ms | Дочерние соты загружают данные | — |
| T+500ms | Переключение (cutover) | — |
| T+600ms | Клиенты переподключены | Игра продолжается, мир разделён |

**Общее время простоя:** ~500-600 мс, что для MMO приемлемо при правильной коммуникации с игроком (не выглядит как баг, а как "расширение мира").

## Ключевые технические детали

1. **Консистентность:** Используется Redis для координации (распределённые блокировки) + NATS для доставки миграционных данных.
2. **Атомарность:** Двухфазный коммит через Grid Manager гарантирует, что либо все 4 соты активируются, либо ни одной.
3. **Детерминизм:** Распределение сущностей по координатам гарантирует, что при повторном сплите (если откат и повтор) сущности попадут в те же самые соты.
4. **Мониторинг:** Каждый сплит логируется с метриками времени, чтобы оптимизировать пороги.

---

Если хочешь, следующим шагом можем рассмотреть **слияние (merge)** — обратный процесс, когда нагрузка падает и 4 соты объединяются обратно в одну, или **рекурсивное углубление** (когда одна из дочерних сот снова делится, создавая уровень 2).
